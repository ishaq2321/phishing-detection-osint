\section{Problem Statement}
The problem addressed by this application is the rapid detection and explanation of phishing attacks. End-users and organizations frequently encounter suspicious URLs, emails, or text snippets, but lack the technical expertise to determine their legitimacy. PhishGuard provides a simple, intuitive web interface where users can submit these artifacts and receive an immediate, understandable threat analysis report, protecting them from malicious social engineering campaigns.

\section{Methods Used}
To protect the user, the application utilizes a hybrid Machine Learning (ML), Open Source Intelligence (OSINT), and Natural Language Processing (NLP) architecture. When a user submits an artifact, the system automatically parses the input, compares it against known global threat databases, and runs mathematical feature models to predict malicious intent. The results are translated into a simple risk score and an explainable breakdown for the user.

\section{User Guide and Interface}

The PhishGuard user interface is designed to be highly accessible and visually intuitive. The core user flow involves accessing the dashboard, inputting a suspicious artifact, and reviewing the generated risk report. No technical expertise or installation is required by the end-user, as the platform is accessed entirely via a standard web browser.

\subsection{Step 1: The Dashboard and Input Submission}\label{dashboard-input}

Upon navigating to the PhishGuard application, the user is presented with a clean, distraction-free dashboard (Figure 2.1). The primary interactive element is the central input field. 

To initiate a security scan, the user simply pastes the suspicious content into this field. The system is designed to be modality-agnostic; the user does not need to specify whether the input is a URL, an email address, or a block of text. The application will automatically detect the format upon submission.

Once the content is pasted, the user clicks the ``Analyze'' button. During the analysis phase, which typically takes between 1 to 5 seconds depending on external network lookups, the interface displays a loading state to assure the user that the request is being processed.

\subsection{Step 2: Interpreting the Threat Analysis Results}\label{interpreting-results}

When the backend orchestrator completes its evaluation, the interface seamlessly transitions to the Threat Analysis Results view (Figure 2.2). This page is structured to provide an immediate, high-level verdict followed by detailed, transparent metrics.

\subsubsection{The Threat Gauge and Verdict}\label{threat-gauge-verdict}

At the top of the results page, the user will see a large, color-coded Threat Gauge. This is the most critical piece of information for non-technical users. It displays an overall risk percentage (from 0\% to 100\%) and a clear textual verdict:
\begin{itemize}
    \item \textbf{Safe (Green):} The content shows no signs of malicious intent or known blacklisting.
    \item \textbf{Suspicious (Amber):} The content exhibits anomalous characteristics (such as a newly registered domain or unusual routing) but is not definitively malicious. Caution is advised.
    \item \textbf{Dangerous (Red):} The content is highly likely to be a phishing attempt, confirmed by threat intelligence databases or strong heuristic matches. Do not interact with the original link.
\end{itemize}

\subsubsection{Detailed Metric Breakdown}\label{detailed-metric-breakdown}

Below the primary gauge, the user can review specific findings that contributed to the final score. This section translates complex technical data into plain-language warnings. For example, it will clearly state if a domain was registered very recently, if the SSL certificate is invalid, or if the URL structure attempts to spoof a known brand. This level of transparency empowers the user to understand \textit{why} a specific verdict was reached, fostering trust in the platform's analysis.
