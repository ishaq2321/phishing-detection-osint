\chapter{User Documentation}
\label{user-documentation}

\section{Problem Statement}

Phishing attacks remain one of the most prevalent cybersecurity threats, exploiting human trust through deceptive URLs, emails, and text messages. The problem addressed by PhishGuard is the rapid detection and transparent explanation of phishing attacks for end-users who lack technical expertise. The application provides a web-based interface where users submit suspicious content and receive an immediate, understandable threat analysis report, protecting them from malicious social engineering campaigns.

\section{Methods Used}

PhishGuard employs a hybrid architecture combining three analytical methods:

\begin{enumerate}
\item \textbf{Machine Learning (XGBoost):} A gradient-boosted decision tree model trained on 150,391 URLs with 21 engineered features (17 lexical + 4 OSINT). The model predicts a phishing probability score between 0.0 and 1.0.
\item \textbf{Open-Source Intelligence (OSINT):} Real-time queries to WHOIS registries, DNS servers, VirusTotal, and AbuseIPDB provide external context about domain age, infrastructure validity, and threat reputation.
\item \textbf{Natural Language Processing (NLP):} A spaCy-based pipeline detecting social engineering tactics such as urgency, brand impersonation, threat warnings, and credential solicitation within unstructured text.
\end{enumerate}

These signals are combined using content-specific weighting: for URL inputs, the ML model contributes 85\% and NLP 15\%; for email and text inputs, NLP contributes 55\%, URL features 25\%, and OSINT 20\%. The final score maps to one of four threat levels: Safe, Suspicious, Dangerous, or Critical.

\section{Accessing the Application}

PhishGuard is a web application requiring no installation. The user accesses the platform through a standard web browser (Chrome, Firefox, Safari, or Edge) by navigating to the deployed URL. The interface is responsive, adapting its layout across desktop, tablet, and mobile devices. The application supports both light and dark themes, defaulting to the system preference. The user can toggle the theme via the header dropdown or the Settings page.

\subsection{The Dashboard}

Upon navigating to the application, the user is presented with the Dashboard (Figure \ref{fig:dashboard}). The Dashboard also displays three capability cards describing the system's detection modes: URL Analysis, OSINT Enrichment, and NLP Detection.

\subsection{Single Analysis}

The Analyse page is the primary interface for submitting content (Figure \ref{fig:analyze-page}). The user performs the following steps:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/analyze-page.png}
\caption{The Analyse page showing input type selector and content field}
\label{fig:analyze-page}
\end{figure}

\begin{enumerate}
\item \textbf{Select the input type} using the dropdown selector at the top of the form. Three modes are available:
\begin{itemize}
\item \textbf{URL:} For analysing a single web address (e.g., \texttt{https://example.com/login}).
\item \textbf{Email:} For pasting the full body of a suspicious email. Two additional optional fields appear: Subject and Sender.
\item \textbf{Text:} For any unstructured text content.
\end{itemize}
\item \textbf{Enter the content} in the input field. In URL mode, this is a single-line input; in Email and Text modes, a multi-line textarea is provided.
\item \textbf{Submit the analysis} by clicking the ``Analyse'' button. A five-step progress indicator is displayed during processing:
\begin{enumerate}
\item Extracting URL features
\item Running OSINT checks
\item Analysing text with NLP
\item Calculating final score
\item Analysis complete
\end{enumerate}
\item \textbf{Cancel if needed} by clicking the ``Cancel'' button that replaces the Analyse button during processing. This aborts the network request.
\end{enumerate}

The Analyse page also includes a tips card advising the user to enter full URLs including the protocol (\texttt{https://}), paste complete email bodies, and a link to the Batch Analysis mode for multiple URLs.

\subsection{Batch Analysis}

For analysing multiple URLs simultaneously, the user navigates to the Batch Analysis page via the link on the Analyse page or the sidebar (Figure \ref{fig:batch-page}). This page allows the user to:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/batch-page.png}
\caption{Batch Analysis page with URL input and results table}
\label{fig:batch-page}
\end{figure}

\begin{itemize}
\item \textbf{Enter up to 50 URLs} in a multi-line text area, one URL per line.
\item \textbf{Upload a file} (\texttt{.txt} or \texttt{.csv}) containing URLs by clicking the ``Upload File'' button.
\item \textbf{Start the analysis} by clicking ``Analyse N URLs.'' The system processes 3 URLs concurrently and displays a progress bar.
\item \textbf{Review the results table} showing each URL's status, threat level, confidence score, and analysis time.
\item \textbf{Export the results} as CSV or JSON files using the export buttons above the table.
\end{itemize}

\subsection{Verdict Banner}

At the top of the results page, a prominent colour-coded banner displays the final verdict. It shows a large shield icon (green for safe, amber for suspicious, red for dangerous, violet for critical), the classification label (``PHISHING'' or ``SAFE''), the animated confidence score as a percentage, the threat level badge, and a plain-language recommendation. The four threat levels are defined in Table \ref{tab:threat_levels}.

\begin{table}[htbp]
\centering
\caption{Threat Level Definitions and Recommendations}
\label{tab:threat_levels}
\begin{tabular}{llll}
\toprule
\textbf{Threat Level} & \textbf{Score Range} & \textbf{Colour} & \textbf{Recommendation} \\
\midrule
Safe & 0.00--0.29 & Green & No action needed \\
Suspicious & 0.30--0.49 & Amber & Exercise caution \\
Dangerous & 0.50--0.69 & Red & Likely phishing, do not interact \\
Critical & 0.70--1.00 & Violet & Confirmed threat, block and report \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Risk Indicators}

Below the verdict, the ``Risk Indicators'' card lists each factor that contributed to the final score (Figure \ref{fig:risk-indicators}). Each indicator is accompanied by a context-aware icon: a lightning bolt for urgency indicators, a theatre mask for brand impersonation, a key for credential solicitation, a link for URL-related findings, and a globe for OSINT-derived indicators. Examples of risk indicators include:

\begin{itemize}
\item ``Domain registered fewer than 30 days ago''
\item ``Uses fear tactics and account suspension threats''
\item ``URL contains suspicious top-level domain''
\item ``No valid DNS records found''
\item ``Brand impersonation detected: PayPal''
\end{itemize}

\subsection{OSINT Intelligence}

The OSINT Intelligence section presents the external intelligence gathered about the analysed domain (Figure \ref{fig:osint-results}). It is organised into three cards:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/osint-results.png}
\caption{OSINT Intelligence section showing domain, DNS, and reputation data}
\label{fig:osint-results}
\end{figure}

\begin{itemize}
\item \textbf{Domain Information:} Displays the domain name, domain age (classified as ``Very New'' if under 30 days, ``New'' if under 180 days, or ``Established'' otherwise), registrar name, and WHOIS privacy status (``Protected'' or ``Public''). Each field includes an info tooltip explaining its relevance.
\item \textbf{DNS Status:} Indicates whether the domain has valid DNS records, shown with a checkmark or cross icon and an explanatory description.
\item \textbf{Reputation:} Displays a reputation score between 0 and 1 on a colour-coded progress bar (green for Good $\geq$0.7, amber for Fair $\geq$0.4, red for Poor), and a blacklist status badge (``Clean'' or ``Blacklisted'').
\end{itemize}

If OSINT data could not be retrieved (e.g., due to API rate limits or timeouts), the section displays a card explaining that OSINT data is not available and the analysis was performed using URL and NLP features only.

\subsection{Feature Extraction and Detected Tactics}

The Feature Extraction section displays four animated counters: the number of suspicious URL features detected, text features detected, OSINT risk indicators, and the total risk indicator count. The Detected Tactics card shows colour-coded badges for each social engineering tactic identified by the NLP pipeline. The ten detectable tactics are: urgency, authority impersonation, brand impersonation, credential request, threat warning, emotional manipulation, monetary request, attachment malware, link manipulation, and social proof. Each badge includes a tooltip explaining the tactic.

\subsection{Share and Export Actions}

The Results page provides four action buttons for sharing the analysis:
\begin{itemize}
\item \textbf{Copy Summary:} Copies a formatted text report to the clipboard containing the content, verdict, threat level, confidence score, recommendation, and risk indicators.
\item \textbf{Copy JSON:} Copies the raw JSON response from the API.
\item \textbf{Copy Link:} Copies the current page URL.
\item \textbf{Print:} Opens the browser print dialog with a print-optimised layout that hides the navigation sidebar and footer.
\end{itemize}

\section{Analysis History}

The History page (Figure \ref{fig:history-page}) provides a persistent record of all analyses conducted by the user. Data is stored in the browser's local storage and persists across sessions.

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/history-page.png}
\caption{Analysis History page with search, filter, and export capabilities}
\label{fig:history-page}
\end{figure}

\subsection{Viewing and Searching History}

The History page displays a sortable table with the following columns: content (truncated), type (URL, email, or text), threat level (colour-coded badge), confidence score, and date. A search bar allows filtering by content or domain name. A dropdown filter enables filtering by threat level (All Levels, Safe, Suspicious, Dangerous, or Critical). Pagination controls at the bottom allow adjusting the page size (10, 25, or 50 entries).

On mobile devices, the table is replaced by a card-based layout with the same information and action buttons.

\subsection{History Actions}

Each history entry provides three actions via a dropdown menu:
\begin{itemize}
\item \textbf{View Results:} Loads the stored analysis response and navigates to the Results page.
\item \textbf{Re-analyse:} Navigates to the Analyse page with the content and type pre-populated, allowing the user to submit the same content again.
\item \textbf{Delete:} Removes the entry from the history.
\end{itemize}

The ``Clear All'' button at the top of the page deletes all history entries after a confirmation dialog.

\subsection{Exporting History}

The Export dropdown provides two options:
\begin{itemize}
\item \textbf{Export as CSV:} Downloads a \texttt{.csv} file with columns for content, type, threat level, score, and date.
\item \textbf{Export as JSON:} Downloads a \texttt{.json} file containing the full analysis data for each entry.
\end{itemize}

\section{How It Works Page}

The How It Works page explains the system's methodology to non-technical users. It contains six sections:

\begin{enumerate}
\item \textbf{Architecture Overview:} A visual pipeline diagram showing how input flows through text analysis, URL feature extraction, and OSINT enrichment into the XGBoost model, producing a final verdict.
\item \textbf{Text Analysis:} An accordion describing six NLP detection categories: urgency patterns, credential harvesting, brand impersonation, fear and threat tactics, suspicious formatting, and emotional manipulation.
\item \textbf{URL Feature Analysis:} An accordion describing six URL-based detection categories: suspicious TLDs, IP addresses in URLs, subdomain depth, homograph attacks, URL shorteners, and path and query anomalies.
\item \textbf{OSINT Enrichment:} An accordion describing six OSINT data sources: WHOIS domain age, registrar reputation, DNS record validation, VirusTotal score, AbuseIPDB reports, and blacklist membership.
\item \textbf{Scoring Algorithm:} Displays the model metrics (96.5\% accuracy, F1 score 96.4\%, AUC-ROC 99.4\%, PR-AUC 99.5\%) and the weighting formula used for URL inputs: $\text{finalScore} = \text{mlPrediction} \times 0.85 + \text{nlpScore} \times 0.15$.
\item \textbf{Threat Levels:} Four cards explaining each threat level with its score range and recommended action.
\end{enumerate}

\subsection{Responsive Design}

The application adapts its layout based on screen size:
\begin{itemize}
\item \textbf{Desktop (1024px and above):} A full sidebar (240px) with icons and labels for navigation.
\item \textbf{Tablet (768px--1023px):} A collapsed icon-only sidebar rail (56px) with tooltips on hover.
\item \textbf{Mobile (below 768px):} A fixed bottom navigation bar with touch-optimised targets (minimum 44x48 pixels, meeting WCAG 2.1 AA requirements). Tables are replaced with card-based layouts.
\end{itemize}

\subsection{Error Handling}

The application provides clear feedback when errors occur:
\begin{itemize}
\item \textbf{Network errors:} A toast notification displays ``Cannot connect to the analysis server. Is the backend running?'' when the API is unreachable.
\item \textbf{Server errors:} A toast notification displays ``The analysis server encountered an internal error. Please try again.'' for HTTP 500+ responses.
\item \textbf{Validation errors:} If the input is invalid, a toast notification shows ``Invalid input:'' followed by the specific error message from the API.
\item \textbf{Error boundaries:} If a page fails to render, a dedicated error page appears with a ``Try Again'' button and a link back to the Dashboard.
\item \textbf{404 page:} Navigating to a non-existent route displays a custom page with links to the Dashboard and Analyse page.
\end{itemize}

\subsection{Backend Health Indicator}

The application header displays a small coloured dot indicating the backend server status, refreshed every 30 seconds: green for connected, amber for degraded (ML model not loaded), and red for unhealthy.

\subsection{Print Support}

The ``Print'' action on the Results page triggers the browser print dialog with a clean layout that hides the navigation sidebar, footer, and toast notifications, using white backgrounds and black text optimised for paper output.
