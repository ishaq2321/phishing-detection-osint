\chapter{User Documentation}
\label{user-documentation}

\section{Problem Statement}

Phishing attacks remain one of the most prevalent cybersecurity threats, exploiting human trust through deceptive URLs, emails, and text messages. The problem addressed by PhishGuard is the rapid detection and transparent explanation of phishing attacks for end-users who lack technical expertise. The application provides a web-based interface where users submit suspicious content and receive an immediate, understandable threat analysis report, protecting them from malicious social engineering campaigns.

\section{Methods Used}

PhishGuard employs a hybrid architecture combining three analytical methods:

\begin{itemize}
\item \textbf{Machine Learning (XGBoost):} A trained model that analyzes URL structure to predict phishing probability.
\item \textbf{Open-Source Intelligence (OSINT):} Live queries verify domain infrastructure and reputation using external threat intelligence sources.
\item \textbf{Natural Language Processing (NLP):} Text analysis detects social engineering tactics such as urgency, brand impersonation, and credential solicitation.
\end{itemize}

For URL inputs, the ML model receives primary weight. For email and text inputs, NLP receives primary weight. The final score maps to one of four threat levels: Safe, Suspicious, Dangerous, or Critical.

\section{Accessing the Application}

PhishGuard is a web application requiring no installation. The user accesses the platform through a standard web browser (Chrome, Firefox, Safari, or Edge) by navigating to the deployed URL. The interface is responsive, adapting its layout across desktop, tablet, and mobile devices. The application supports light, dark, and system-following themes, defaulting to the operating system preference. The user can switch themes via a three-option dropdown (Light, Dark, System) in the application header, or via the binary toggle on the Settings page.

Upon navigating to the application, the user is presented with the Dashboard (Figure \ref{fig:dashboard}). This screen serves as the landing page and provides two primary call-to-action buttons: ``Analyse Now,'' which navigates to the analysis page, and ``How It Works,'' which navigates to the methodology explanation. The Dashboard also displays three capability cards describing the system's detection modes: URL Analysis, OSINT Enrichment, and NLP Detection. Below the capability cards, a ``System Overview'' card summarises the current scoring weights (e.g., ``text 40\%, URL 25\%, OSINT 35\%''), the number of threat levels (4), and the available input modes (3).

\subsection{Single Analysis}

The Analyse page is the primary interface for submitting content (Figure \ref{fig:analyze-page}). The user performs the following steps:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/analyze-page.png}
\caption{The Analyse page showing input type selector and content field}
\label{fig:analyze-page}
\end{figure}

\begin{enumerate}
\item \textbf{Select the input type} using the dropdown selector at the top of the form. Three modes are available:
\begin{itemize}
\item \textbf{URL:} For analysing a single web address (e.g., \texttt{https://example.com/login}).
\item \textbf{Email:} For pasting the full body of a suspicious email. Two additional optional fields appear: Subject and Sender.
\item \textbf{Text:} For any unstructured text content.
\end{itemize}
\item \textbf{Enter the content} in the input field. In URL mode, this is a single-line input; in Email and Text modes, a multi-line textarea is provided.
\item \textbf{Submit the analysis} by clicking the ``Analyse'' button. A five-step progress indicator is displayed during processing:
\begin{enumerate}
\item Extracting URL features
\item Running OSINT checks
\item Analysing text with NLP
\item Calculating final score
\item Analysis complete
\end{enumerate}
The progress indicator is a client-side animation that runs on a timer to provide visual feedback while the single API request is in flight. Navigation to the Results page occurs automatically once the server response is received.
\item \textbf{Cancel if needed} by clicking the ``Cancel'' button that replaces the Analyse button during processing. This stops the network request in progress.
\item \textbf{Clear the form} using the ``Clear'' ghost-variant button that appears when content has been entered. This resets the content, subject, and sender fields.
\end{enumerate}

If the user navigates to the Analyse page from the History page's ``Re-analyse'' action, the content and input type are automatically pre-populated via URL query parameters (\texttt{/analyze?content=\ldots\&type=\ldots}).

The Analyse page also includes a tips card advising the user to enter full URLs including the protocol (\texttt{https://}), paste complete email bodies, and a link to the Batch Analysis mode for multiple URLs.

\subsection{Batch Analysis}

For analysing multiple URLs simultaneously, the user navigates to the Batch Analysis page via the link on the Analyse page or the sidebar (Figure \ref{fig:batch-page}). This page allows the user to:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/batch-page.png}
\caption{Batch Analysis page with URL input and results table}
\label{fig:batch-page}
\end{figure}

\begin{itemize}
\item \textbf{Enter up to 50 URLs} in a multi-line text area, one URL per line. A live counter displays ``N / 50 URLs'' and turns red with ``(max 50)'' when the limit is exceeded.
\item \textbf{Upload a file} (\texttt{.txt} or \texttt{.csv}) containing URLs by clicking the ``Upload File'' button.
\item \textbf{Clear the input} using a ``Clear'' button that resets the URL text area.
\item \textbf{Start the analysis} by clicking ``Analyse N URLs.'' The system processes 3 URLs concurrently and displays a progress bar. A ``Cancel'' button with a StopCircle icon replaces the Analyse button during processing, aborting pending requests via an AbortController.
\item \textbf{Review the summary statistics} displayed above the results table as colour-coded badges showing the count of Safe, Suspicious, Dangerous, Critical, and Failed entries.
\item \textbf{Review the results table} showing each URL's status badge (Pending, Running, Done, or Error), threat level, confidence score, and analysis time.
\item \textbf{Export the results} as CSV or JSON files using the export buttons above the table.
\end{itemize}

Upon completion, a success toast notification displays ``Batch analysis complete --- N URLs processed.'' If cancelled, an error toast displays ``Batch analysis cancelled.''

\section{Interpreting the Results}

When the backend completes its evaluation, the application navigates to the Results page (Figure \ref{fig:results-page}). This page provides a comprehensive breakdown of the analysis. It is important to note that the Results page is ephemeral: it reads from an in-memory React context, not from persistent storage. If the user refreshes the page or navigates directly to \texttt{/results} via the URL bar, an empty-state card appears with a ``No Results Yet'' title and a ``Go to Analyse'' button. To revisit past analyses, the user should use the History page (Section \ref{sec:history}), which provides persistent storage.

\subsection{Content Preview}

At the top of the results, a Content Preview card displays the analysed content (URLs are shown as clickable external links; text is truncated at 500 characters), the content type badge, the analysis time, and an ``Analysed on'' timestamp. An outline ``New Analysis'' button in the results header allows the user to return to the Analyse page.

\subsection{Verdict Banner}

Below the content preview, a prominent colour-coded banner displays the final verdict. It shows a large shield icon (green for safe, amber for suspicious, red for dangerous, violet for critical), the classification label (``PHISHING'' or ``SAFE''), the animated confidence score as a percentage, the threat level badge, and a plain-language recommendation. \subsection{Risk Indicators}

The ``Risk Indicators (N)'' card lists each factor that contributed to the final score, with the count displayed in the card header (Figure \ref{fig:risk-indicators}). Each indicator is accompanied by a context-aware icon: a lightning bolt for urgency indicators, a theatre mask for brand impersonation, a key for credential solicitation, a link for URL-related findings, a globe for OSINT-derived indicators, and a fallback alert-triangle icon for any indicators that do not match the preceding categories. Examples of risk indicators include:

\begin{itemize}
\item ``Domain registered fewer than 30 days ago''
\item ``Uses fear tactics and account suspension threats''
\item ``URL contains suspicious top-level domain''
\item ``No valid DNS records found''
\item ``Brand impersonation detected: PayPal''
\end{itemize}

\subsection{OSINT Intelligence}

The OSINT Intelligence section presents the external intelligence gathered about the analysed domain (Figure \ref{fig:osint-results}). It is organised into three cards:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/osint-results.png}
\caption{OSINT Intelligence section showing domain, DNS, and reputation data}
\label{fig:osint-results}
\end{figure}

\begin{itemize}
\item \textbf{Domain Information:} Displays the domain name, domain age (classified as ``Very New'' if under 30 days, ``New'' if under 180 days, or ``Established'' otherwise), registrar name, and WHOIS privacy status (``Protected'' or ``Public''). Each field includes an info tooltip explaining its relevance: for example, ``Newly registered domains are more likely to be used for phishing'' (Domain Age) and ``Privacy-protected WHOIS records can hide the true registrant'' (Privacy).
\item \textbf{DNS Status:} Indicates whether the domain has valid DNS records, shown with a checkmark or cross icon. The tooltip explains: ``DNS records indicate whether the domain resolves to a valid server.''
\item \textbf{Reputation:} Displays a reputation score between 0 and 1 on a colour-coded progress bar (green for Good $\geq$0.7, amber for Fair $\geq$0.4, red for Poor), and a blacklist status badge (``Clean'' or ``Blacklisted''). The tooltip explains: ``Aggregated reputation from OSINT sources (WHOIS age, DNS, blacklists)'' and ``Checks whether the domain appears on known phishing/malware blacklists.''
\end{itemize}

If OSINT data could not be retrieved because the input does not contain a recognisable domain, the section displays a card explaining: ``No OSINT enrichment data was collected for this analysis. This can happen when the input does not contain a recognisable domain.''

\subsection{Feature Extraction and Detected Tactics}

The Feature Extraction section displays four animated counters: the number of suspicious URL features detected, text features detected, OSINT risk indicators, and the total risk indicator count. The Detected Tactics card shows colour-coded badges for each social engineering tactic identified by the NLP pipeline, with a destructive count badge in the card header. The ten detectable tactics are: urgency, authority impersonation, brand impersonation, credential request, threat warning, emotional manipulation, monetary request, attachment malware, link manipulation, and social proof. Each badge includes a tooltip explaining the tactic. When no tactics are found, a green positive-state card displays ``No suspicious tactics detected'' with a ShieldCheck icon.

\subsection{Score Visualisation}

The Score Visualisation section provides three interactive charts (Figure \ref{fig:score-viz}), loaded on demand for faster page loading:

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/score-viz.png}
\caption{Score Visualisation showing the ML/NLP breakdown, threat gauge, and confidence bar}
\label{fig:score-viz}
\end{figure}

\begin{itemize}
\item \textbf{Score Breakdown (Donut Chart):} A Recharts donut chart showing the proportion of the ML model contribution (85\% for URL inputs, 55\% for text inputs) versus the NLP contribution, with the final risk score displayed in the centre.
\item \textbf{Threat Gauge:} A semi-circular SVG gauge with four colour zones (green, yellow, orange, red) and an animated needle pointing to the calculated score.
\item \textbf{Confidence Bar:} A horizontal progress bar colour-coded by threat level, with a gradient scale showing the boundaries between threat zones at 30\%, 50\%, 70\%, and 100\%.
\end{itemize}

\subsection{Share and Export Actions}

The Results page provides four action buttons for sharing the analysis, each with a tooltip describing its function:
\begin{itemize}
\item \textbf{Copy Summary:} Copies a formatted text report to the clipboard containing the content, verdict, threat level, confidence score, recommendation, and risk indicators. Upon success, the button icon changes to a green checkmark and the label changes to ``Copied!'' for 2 seconds, accompanied by a success toast.
\item \textbf{Copy JSON:} Copies the raw JSON response from the API, with the same visual feedback and toast notification.
\item \textbf{Copy Link:} Copies the current page URL, with the same visual feedback and toast notification.
\item \textbf{Print:} Opens the browser print dialog with a print-optimised layout that hides the navigation sidebar, footer, mobile navigation, sheet overlays, and toast notifications, using white backgrounds and black text optimised for paper output.
\end{itemize}

If the browser's Clipboard API is unavailable, a fallback copy method is used to maintain compatibility with older browsers. If clipboard access fails entirely, an error toast displays ``Failed to copy --- Your browser may not support clipboard access.''

\section{Analysis History}
\label{sec:history}

The History page (Figure \ref{fig:history-page}) provides a persistent record of all analyses conducted by the user. The system maintains dual history stores: a backend in-memory store for internal logging, and a frontend localStorage store for user-facing persistence across sessions. The page subtitle displays the total count: ``N analysis result(s) stored locally.'' When no analyses have been run yet, a ``No Analyses Yet'' card appears with a ``Start Analysing'' link button. If the auto-clear setting is enabled, entries older than the configured threshold are automatically removed when the History page is loaded.

\subsection{Viewing and Searching History}

The History page displays a sortable table with the following columns: content (truncated), type (URL, email, or text), threat level (colour-coded badge), confidence score, and date. A search bar allows filtering by content or domain name. A dropdown filter enables filtering by threat level (All Levels, Safe, Suspicious, Dangerous, or Critical). Pagination controls at the bottom include first-page, previous-page, next-page, and last-page buttons, along with a page size selector (10, 25, or 50 entries) and a filtered result count (``N result(s)'') reflecting the current search and filter state.

On mobile devices, the table is replaced by a card-based layout with the same information and action buttons.

\subsection{History Actions}

Each history entry provides three actions via a dropdown menu:
\begin{itemize}
\item \textbf{View Results:} Loads the stored analysis response and navigates to the Results page.
\item \textbf{Re-analyse:} Navigates to the Analyse page with the content and type pre-populated via URL query parameters, allowing the user to submit the same content again.
\item \textbf{Delete:} Removes the entry from the history.
\end{itemize}

The ``Clear All'' button at the top of the page deletes all history entries after a confirmation dialog displaying the exact count: ``This will permanently delete all N analysis result(s) from your local storage. This action cannot be undone.''

\subsection{Exporting History}

The Export dropdown provides two options, each triggering an info toast notification upon completion:
\begin{itemize}
\item \textbf{Export as CSV:} Downloads a \texttt{.csv} file with columns for content, type, threat level, score, and date. An info toast displays ``Exported as CSV.''
\item \textbf{Export as JSON:} Downloads a \texttt{.json} file containing the full analysis data for each entry. An info toast displays ``Exported as JSON.''
\end{itemize}

\section{How It Works Page}

The How It Works page explains the system's methodology to non-technical users. It contains six sections:

\begin{enumerate}
\item \textbf{Architecture Overview:} A visual pipeline diagram showing how input flows through text analysis, URL feature extraction, and OSINT enrichment into the XGBoost model, producing a final verdict.
\item \textbf{Text Analysis:} An accordion describing six NLP detection categories: urgency patterns, credential harvesting, brand impersonation, fear and threat tactics, suspicious formatting, and emotional manipulation.
\item \textbf{URL Feature Analysis:} An accordion describing six URL-based detection categories: suspicious TLDs, IP addresses in URLs, subdomain depth, homograph attacks, URL shorteners, and path and query anomalies.
\item \textbf{OSINT Enrichment:} An accordion describing six OSINT data sources: WHOIS domain age, registrar reputation, DNS record validation, VirusTotal score, AbuseIPDB reports, and blacklist membership.
\item \textbf{Scoring Algorithm:} Explains how the ML model receives primary weight for URL inputs and NLP receives primary weight for text inputs. The exact weighting is detailed in Chapter \ref{chapter-7-scoring-and-classification}.
\item \textbf{Threat Levels:} Four cards explaining each threat level with its score range and recommended action.
\end{enumerate}

At the bottom of the page, a ``Ready to Try It?'' call-to-action section provides a ``Start Analysing'' link button back to the Analyse page.

\section{Settings Page}

The Settings page allows the user to configure the application (Figure \ref{fig:settings-page}). All settings changes take effect immediately upon modification --- there is no explicit ``Save'' button. Settings are persisted in the browser's local storage.

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\linewidth,height=0.65\textheight,keepaspectratio]{images/settings-page.png}
\caption{Settings page with API configuration, display preferences, and history management}
\label{fig:settings-page}
\end{figure}

\begin{itemize}
\item \textbf{API Configuration:} The Backend URL input field specifies the address of the PhishGuard API server. Below the input, the default value is displayed: ``Default: \texttt{http://localhost:8000}.'' The ``Test'' button verifies connectivity; while testing, the button displays a spinning loader icon and the text ``Testing\ldots'' and is disabled. The status badge next to the button shows four states: no badge (idle, before any test), ``Testing'' (during the request), ``Connected'' (on success, accompanied by a success toast), or ``Failed'' (on failure, accompanied by an error toast).
\item \textbf{Display Preferences:} A dark mode toggle switches between light and dark themes on the Settings page, while the header dropdown offers a three-way choice of Light, Dark, or System (following the operating system preference). A detail level selector adjusts the verbosity of result displays (Simple, Detailed, or Expert).
\item \textbf{History Management:} The maximum number of history entries can be set to 10, 25, 50, or 100. An auto-clear dropdown automatically removes entries older than a configured period: Never (the default), 7 days, 14 days, or 30 days. A ``Clear History'' destructive-variant button immediately removes all entries after a confirmation dialog; the button displays the current count (``N entry/entries stored'') and is disabled when no entries exist. A success toast confirms the action.
\item \textbf{About:} Displays application metadata including the application name, tagline (``OSINT-Enhanced Phishing Detection''), version (1.0.0), author, supervisor, and university.
\end{itemize}

A ``Reset to Defaults'' button at the bottom restores all settings to their original values, accompanied by a warning toast notification confirming the reset.

Pressing ``?'' opens a shortcuts help dialog listing all seven shortcuts with formatted key badges; on macOS, the ``Ctrl'' labels are automatically replaced with the Command symbol ($\backslash$cmd) in the dialog.

\subsection{Responsive Design}

The application adapts its layout based on screen size:
\begin{itemize}
\item \textbf{Desktop (1024px and above):} A full sidebar (240px) with icons and labels for navigation. The sidebar footer displays ``PhishGuard'' and ``v1.0.0 . ELTE FI.''
\item \textbf{Tablet (768px--1023px):} A collapsed icon-only sidebar rail (56px) with tooltips on hover. The rail footer shows ``v1.0.0.''
\item \textbf{Mobile (below 768px):} The sidebar is hidden; tapping the hamburger icon (44x44px touch target) in the header opens a slide-in Sheet drawer from the left containing the full navigation. A fixed bottom navigation bar replaces the sidebar with touch-optimised targets (minimum 44x48 pixels, meeting WCAG 2.1 AA requirements). Tables are replaced with card-based layouts.
\end{itemize}

An application footer at the bottom of the content area displays ``PhishGuard \textcopyright{} 2026 . Ishaq Muhammad (PXPRGK) . ELTE Faculty of Informatics.''

\subsection{Loading States}

The application provides two types of loading indicators during route transitions:
\begin{itemize}
\item \textbf{Full-page spinner:} When loading a route segment for the first time, an animated shield icon with ``Loading\ldots / Preparing your workspace'' text is displayed. The spinner includes \texttt{role="status"} and \texttt{aria-label="Loading page"} for screen reader compatibility.
\item \textbf{App-shell skeleton:} Within the application shell (sidebar and header visible), route transitions show a skeleton placeholder mimicking the page layout with animated heading and card placeholders.
\end{itemize}

\subsection{Toast Notifications}

The application uses four types of toast notifications to provide feedback:
\begin{itemize}
\item \textbf{Success (green):} Analysis complete, connection successful, history cleared, settings reset, clipboard copied.
\item \textbf{Error (red):} Analysis failed, connection failed, batch cancelled, clipboard copy failed.
\item \textbf{Warning (amber):} Settings reset confirmation.
\item \textbf{Info (blue):} Export actions completed (CSV, JSON).
\end{itemize}

All toast notifications appear in the bottom-right corner and auto-dismiss after a configurable duration.

\subsection{Error Handling}

The application provides clear feedback when errors occur:
\begin{itemize}
\item \textbf{Network errors:} A toast notification displays ``Cannot connect to the analysis server. Is the backend running?'' when the API is unreachable.
\item \textbf{Server errors:} A toast notification displays ``The analysis server encountered an internal error. Please try again.'' for HTTP 500+ responses.
\item \textbf{Validation errors:} If the input is invalid, a toast notification shows ``Invalid input:'' followed by the specific error message from the API.
\item \textbf{App-shell error boundary:} If a page fails to render within the application shell (sidebar and header remain visible), an error page appears with the error message and three navigation buttons: ``Try Again,'' ``Dashboard,'' and ``Analyse.''
\item \textbf{Global error boundary:} If a catastrophic error occurs outside the app shell, a full-screen error page appears (no sidebar) with the error message, an error digest hash for debugging, a ``Try Again'' button, and a ``Go to Dashboard'' link.
\item \textbf{404 page:} Navigating to a non-existent route displays a custom page with a large ``404'' heading, a ShieldOff icon, an explanation message, and two buttons: ``Go to Dashboard'' (primary) and ``Analyse Content'' (outline).
\end{itemize}

\subsection{Backend Health Indicator}

The application header displays a small coloured dot indicating the backend server status, refreshed every 30 seconds. Four states are possible: muted grey (``Checking backend\ldots,'' before the first health check completes), green for connected, amber for degraded (ML model not loaded), and red for unhealthy. On screens $\geq$sm, a text label appears alongside the dot; on smaller screens, an \texttt{sr-only} label ensures screen reader compatibility. The health indicator uses \texttt{role="status"} so that screen readers announce status changes.

\subsection{Print Support}

The ``Print'' action on the Results page triggers the browser print dialog with a clean layout that hides the navigation sidebar, footer, mobile navigation, Sheet overlays, and toast notifications, resetting card backgrounds and borders for ink-saving output. All colour and background transitions are globally animated over 200 milliseconds for smooth theme switching, and are automatically disabled when the user's operating system preference is set to reduced motion.

\subsection{Accessibility}

The application implements several WCAG 2.1 accessibility features to ensure it is usable by people with diverse abilities:
\begin{itemize}
\item \textbf{Screen reader support:} Users can navigate the application using screen readers, with the current page and status announced.
\item \textbf{Keyboard navigation:} All interactive elements are accessible via keyboard, with visible focus indicators.
\item \textbf{Reduced motion support:} Animations are automatically disabled when the user's operating system preference is set to reduce motion.
\item \textbf{Mobile accessibility:} Touch targets meet WCAG 2.1 AA requirements (minimum 44x48 pixels).
\end{itemize}
